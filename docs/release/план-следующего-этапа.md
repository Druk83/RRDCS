# План следующего этапа развития (Next Stage Plan)

## 1. Назначение
Документ фиксирует следующий этап развития RRDCS после закрытия MVP: расширение на несколько репозиториев, усиление политик качества и повышение операционной зрелости.

## 3. Цель этапа
Перейти от одного пилотного репозитория и MVP-политик к устойчивой multi-repo модели, где:
- подключение новых репозиториев происходит повторяемо;
- часть audit-only контуров может быть повышена до blocking gate;
- качество evidence и наблюдаемость процесса становятся частью эксплуатации;
- политика и toolchain остаются управляемыми через versioned profiles.

## 4. Объем следующего этапа

### 4.1 Многорепозиторная раскатка (Multi-repo rollout)
- Подготовить схему подключения не одного, а нескольких целевых репозиториев.
- Формализовать последовательность раскатки по группам репозиториев.
- Описать критерии перехода от pilot к wave и далее к global.
- Определить первую волну репозиториев (pilot set).
- Первый пилотный набор (pilot set) начинается с текущего пилотного репозитория; дополнительные репозитории подключаются только если они используют ту же базовую схему `policy/profile/workflow` и не требуют ручной миграции.
- Кандидат в пилотный набор (pilot set) должен иметь:
  - стабильный PR-flow;
  - тот же набор core checks или совместимый поднабор;
  - понятный rollback и владельца;
  - отсутствие уникальных исключений, не описанных в runbook;
  - готовность принимать audit/required transition без изменения core scripts.

### 4.2 Усиление политик (hardening)
- Перевести `coverage` из audit-only в блокирующий гейт (blocking gate) там, где метрика стабилизирована.
- Пересмотреть порог и поведение `small PR policy`.
- Оценить, какие проверки из security/style стоит сделать обязательными по умолчанию.
- Зафиксировать, какие policies остаются audit-only до следующего решения.
- Для stage-2 hardening по умолчанию рассматриваются только политики, которые уже имеют воспроизводимый отчет и понятный rollback.
- Базовый порядок перевода: `audit-only` -> `pilot blocking` -> `global blocking`.
- На старте stage-2 первыми кандидатами на перевод считаются `coverage` и политика маленьких PR (small PR policy); security/style остаются обязательными в MVP-контуре, но не расширяются без отдельного решения.
- Для `coverage` пилотный blocking-порог соответствует `M2` в `docs/requirements/тестирование/coverage-roadmap.md`, то есть `line coverage >= 55%`.
- Для `small PR policy` ориентир stage-2 остается измеримым: code PR с `churn > 100 changed lines` получает warning, а не блокировку, пока не подтвержден пилотный переход в blocking.
- Каждое решение о переводе политики должно ссылаться на пилотные метрики, evidence и rollback-план.

### 4.3 Операционная зрелость
- Добавить метрики стабильности, повторяемости и времени прохождения проверок.
- Зафиксировать минимальный набор операционных сигналов (operational signals) для stage-2:
  - медианное время прохождения core pipeline;
  - доля прогонов с полным evidence package;
  - доля ложных блокировок, снятых после повторного прогона;
  - время возврата в `audit` при инциденте;
  - количество ручных вмешательств в pipeline.
- Формализовать rollback playbook для multi-repo сценария.
- Усилить контроль качества evidence и отчетов.
- Для каждого failed check обязателен `reason`, `log_ref` и ссылка на артефакт.
- Отчеты stage-2 должны позволять восстановить последовательность событий без обращения к внешнему контексту.
- Формат публикации operational signals закреплен в `docs/requirements/тестирование/испытание системы.md`.

### 4.4 Поддержка изменений (Change Management)
- Свести изменения политики, профиля и workflow к единому процессу управления изменениями (release/change management).
- Подготовить правила совместимости между версиями профилей и workflow.
- Описать правила для ломающих (breaking) и неломающих (non-breaking) изменений.
- Ввести матрицу совместимости `policy/profile/workflow` для подключенных репозиториев.
- Ломающие изменения (breaking changes) допускаются только через pilot-подтверждение и отдельное утверждение (approval).
- Неломающие изменения (non-breaking changes) могут идти через обычный wave/global-процесс после успешного review.
- Каждое изменение должно иметь запись в заметке о релизе (release note) и ссылку на план отката (rollback-plan).
- Ломающие изменения (breaking change) для policy/profile включают:
  - изменение набора required checks;
  - изменение смысла `enforcement_mode`;
  - изменение входных параметров reusable workflow;
  - изменение формата evidence или обязательных артефактов;
  - любое изменение, которое требует ручной миграции подключенных репозиториев.
- Неломающие изменения (non-breaking change) включают:
  - исправления текста и метаданных;
  - обновление документации;
  - patch-фиксы без изменения контракта;
  - добавление необязательных проверок, не влияющих на current required set.
- Отдельная схема закрепления версий (version pinning) для `wave/global` не вводится; pinning остается общим, а различается только маршрут approval и раскатки.

#### Матрица совместимости
| Тип изменения | policy/profile | workflow | Канал раскатки | Утверждение |
|---|---|---|---|---|
| Non-breaking policy/profile patch | тот же major, новый patch | совместимый patch/minor | pilot / wave / global | обычный review |
| Non-breaking workflow patch | тот же major | совместимый patch/minor | pilot / wave / global | обычный review |
| Breaking policy/profile change | новый major | проверенный совместимый workflow | pilot сначала, затем wave/global | отдельный approval |
| Breaking workflow change | возможно новый major | новый major или несовместимый patch | pilot сначала, затем wave/global | отдельный approval |

- Любое изменение из таблицы должно иметь ссылку на заметку о релизе (release note) и план отката (rollback-plan).
- Для перехода в `wave` и `global` требуется подтверждение, что пилотная поставка стабильна и evidence воспроизводим.
- Минимальные критерии перехода `pilot -> wave`:
  - подтверждено несколько успешных прогонов подряд без ручных обходов;
  - evidence публикуется полностью и воспроизводится;
  - rollback на pilot проверен;
  - нет открытых блокирующих инцидентов.
- Минимальные критерии перехода `wave -> global`:
  - пилотная группа стабильна;
  - политика совместима с workflow без ручных исключений;
  - нет устойчивых ложных блокировок;
  - approval подтвержден владельцем политики и техническим лидером.

## 5. Критерии входа
- MVP полностью закрыт и подтвержден.
- `gap-tracking` не содержит открытых пробелов.
- Существуют актуальные артефакты `docs/release/план-релиза-мвп.md`, `docs/operations/базовый-операционный-контур-гитхаб.md` и `.out/`.
- Есть подтвержденные данные по pilot-репозиторию и текущему набору checks.

## 6. Критерии успеха
- Multi-repo onboarding воспроизводится без ручных исключений.
- Расширение policy не ломает текущий `audit -> required` контур.
- Blocking gates становятся настраиваемыми по репозиториям или группам репозиториев.
- Отчеты и evidence остаются стабильными и пригодными для проверки.
- Для пилотной группы репозиториев есть подтвержденный путь перевода из audit-only в blocking.
- Для каждого stage-2 инцидента можно показать причину, лог и артефакт восстановления.
- Для подключенных репозиториев есть актуальная матрица совместимости версий.

## 7. Риски
- Слишком раннее ужесточение coverage может создать ложные блокировки.
- Унификация политик между репозиториями может потребовать больше исключений, чем ожидается.
- Multi-repo rollout без зрелого rollback увеличивает стоимость инцидентов.
- Усложнение workflow может ухудшить время обратной связи для разработчиков.

## 8. Рекомендуемый порядок работ
1. Зафиксировать целевой scope stage-2 и список обязательных улучшений.
2. Подготовить задачи на multi-repo rollout и hardening.
3. Согласовать метрики, по которым `audit-only` контуры могут быть повышены до blocking.
4. Обновить onboarding runbook и policy/profile соглашения.
5. Провести пилот на ограниченной группе репозиториев.
6. Зафиксировать итоговый список изменений для следующего релиза.
7. Обновить матрицу совместимости перед переходом на следующую волну.

## 9. Критерии выхода
- Подзадачи `A4-01` .. `A4-04` завершены или переведены в следующий план работ без потери трассировки.
- Обновлены `docs/release/план-релиза-мвп.md` и `docs/operations/базовый-операционный-контур-гитхаб.md`, если stage-2 потребовал изменения правил.
- Зафиксирован набор репозиториев для первого пилота stage-2.
- Определен как минимум один blocking gate, который переводится из audit-only на этапе hardening.
- Подготовлен rollback playbook для много-репозиторного сценария.
- Зафиксирован набор operational signals и формат их публикации.
- Подготовлена и актуализирована матрица совместимости `policy/profile/workflow`.

## 10. Связь с текущим состоянием
MVP считается закрытым контуром. Этот документ описывает не исправление пробелов, а следующий уровень развития поверх уже работающей базы.
