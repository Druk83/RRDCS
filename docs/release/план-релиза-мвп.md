# План релиза MVP (MVP Release Plan)

## 1. Назначение
Документ определяет порядок релизной приемки MVP RRDCS: критерии `Go/No-Go`, версионирование, rollout и rollback.

## 2. Связанные документы
- `docs/requirements/требования/требования_внутренний_стандарт.md`
- `docs/requirements/тестирование/испытание системы.md`
- `docs/requirements/матрица-сквозной-трассировки.md`
- `docs/requirements/сценарии/policy-and-toolchain-governance/repository-onboarding-runbook.md`
- `docs/operations/базовый-операционный-контур-гитхаб.md`

## 3. Объем релиза MVP
- Локальная предварительная проверка (`check-all`).
- Шлюзы pull request (PR gates) в GitHub Actions.
- Пакеты `style/security/platform`.
- Публикация доказательств (evidence) (`summary + artifacts`).
- Подключение репозитория через профиль интеграции.
- Режимы внедрения `audit -> required`.

## 4. Критерии входа
- Реализация задач MVP завершена и подтверждена в `.tasks/*`.
- Выполнен полный прогон `T-001...T-011`.
- Настройки защиты ветки (branch protection) и обязательных проверок (required checks) применены.
- Нет открытых блокирующих пробелов в `docs/requirements/gap-tracking.md`.

## 5. Критерии решения о запуске (Go/No-Go)

### 5.1 Запуск (Go)
- Все обязательные требования (MUST) уровня FR/NFR подтверждены.
- Тесты `T-001...T-011` имеют статус `pass` или документированное допустимое отклонение с ответственным (owner) и сроком исправления.
- Шлюз слияния корректно блокирует `merge` при `failed` обязательных проверках.
- Сценарий подключения нового репозитория подтвержден в режимах `audit` и `required`.

### 5.2 Не запускать (No-Go)
- Любой критерий `MUST` не выполнен.
- Нарушена надежность шлюза слияния (ложный допуск изменений в защищенную ветку).
- Нет воспроизводимых доказательств (evidence) для проваленных сценариев.
- Отсутствует согласованный план отката (rollback) для актуальной версии policy/profile.

## 6. Версионирование релиза
- Версия релиза: `vMAJOR.MINOR.PATCH`.
- Для MVP рекомендуется `v1.0.0`.
- Версии policy/profile (политики/профиля) фиксируются отдельно и должны быть связаны с релизной версией в заметке о релизе (release note).
- Для первой production-версии используется один релизный тег `v1.0.0`; отдельный tag для policy/profile не вводится.

## 6.1 Состав первой раскатки (rollout)
- Первая раскатка (rollout) выполняется на одном пилотном репозитории: текущем репозитории-носителе policy/profile.
- Каналы `wave` и `global` для первой раскатки (rollout) остаются пустыми и подключаются только после стабилизации пилота.

## 7. Порядок релиза
1. Заморозить изменения policy/profile и required checks на время приемки.
2. Выполнить финальный приемочный прогон `T-001...T-011`.
3. Сформировать отчет по верификации (verification report) со ссылками на доказательства (evidence).
4. Провести review решения `Go/No-Go` (Tech Lead + Owner policy).
5. При `Go` зафиксировать релизную версию и открыть раскатку (rollout).
6. Выполнить раскатку (rollout), начиная с одного пилотного репозитория; каналы `wave` и `global` подключать после стабилизации пилота (если применимо).

## 7.1 Политика gate-ов на MVP
- Блокирующие проверки на первом проходе: PR gates, security/SAST, style checks и architecture guard.
- Только для аудита (audit-only) на первом проходе: coverage и small PR policy.
- Если контур только для аудита (audit-only) падает, он не блокирует merge на MVP, но evidence и summary должны публиковаться.

## 8. Процедура отката (Rollback Procedure)
- Триггеры отката (rollback):
  - рост блокирующих ложных срабатываний
  - нестабильность обязательных проверок (required checks)
  - регресс логики шлюза слияния (merge-gate)
- Действия:
  1. Перевести затронутые репозитории в режим `audit`.
  2. Вернуть предыдущую стабильную версию policy/profile.
  3. Зафиксировать инцидент, причину и корректирующие действия.
  4. Повторить приемочные тесты перед повторным переходом в `required`.

## 9. Артефакты релиза
- Отчет по верификации (verification report) с результатами `T-001...T-011`.
- Ссылки на прогоны CI (CI runs) и артефакты (artifacts).
- Версия policy/profile и версия релиза.
- Решение `Go/No-Go` с ответственными.
- План rollout и, при необходимости, протокол rollback.
