# Файловая структура проекта (File Structure) — RRDCS

## 0. Контекст
- **Проект:** RRDCS
- **Дата обновления:** 2026-04-03
- **Основание:** `.project-structure.json`, `docs/requirements/архитектура/описание TL.md`

## 1. Общая структура репозитория (as-is)

```text
RRDCS/
├── .approach/                   # Подходы и методологии по этапам SDD
├── .chatlog/                    # Логи сессий чата (по запросу)
├── .claude/                     # Конфигурация агентов (локально)
├── .diagrams/                   # Служебные диаграммы процесса разработки
├── .github/                     # Репозиторные настройки (workflows пока не добавлены)
├── .issues/                     # Реестр проблем/багов
├── .manifest/                   # Манифесты правил (H2)
├── .requirements/               # Шаблоны/методики (read-only)
├── .sources/                    # Исходные материалы проекта
├── .tasks/                      # Реестр задач и PDD
├── .tools/                      # Скрипты и утилиты проекта
├── docs/                        # Результаты документации (H8)
├── AGENTS.md                    # H0-правила для агента
├── README.md                    # L0-обзор проекта
└── .project-structure.json      # Каноническая карта структуры
```

## 2. Детализация ключевых каталогов

| Каталог | Назначение | Статус |
|---|---|---|
| `.manifest/` | Правила и политики (SDD, GAP, tools, dependencies, hierarchy) | must |
| `.requirements/` | Шаблоны этапов трека (методика, не редактируется как результат) | must |
| `docs/requirements/` | Фактические результаты этапов [1]-[9] | must |
| `.tools/pdd/` | Инструмент сканирования `@todo` и генерации реестра | must |
| `.tools/plantuml-render/` | Рендер `.plantuml` в PNG/SVG через Kroki | must |
| `.github/workflows/` | CI/CD workflows для PR gates и reusable integration workflows | planned |
| `src/`, `tests/` | Исходный код системы и автотесты (если появятся) | optional |

## 3. Соответствие архитектурным слоям

| Архитектурный слой | Где в репозитории | Комментарий |
|---|---|---|
| Business Layer (Бизнес-слой) | `docs/requirements/сценарии/`, `docs/requirements/домены/` | Модель процессов, мероприятий и контекстов |
| Application Layer (Прикладной слой) | `.tools/`, `.github/workflows/` | Логика проверок и оркестрация исполнения |
| Technology Layer (Технологический слой) | `.tools/plantuml-render/docker-compose*.yml`, policy docs | Runtime/infra-контракты и окружение запуска |

## 5. Профиль структуры (Repository Profile)
- **Текущий профиль:** документационно-инструментальный репозиторий (Documentation + Tooling Repo).
- **Целевой профиль после внедрения PR-gates:** репозиторий с CI-оркестрацией (CI-Orchestrated Repo), где `.github/workflows/*` вызывает `.tools/*`.
- **Целевой профиль multi-repo:** подключение внешних репозиториев через `Repository Integration Profile` и reusable workflow без дублирования логики.

## 6. Конвенции именования
- Каталоги и файлы скриптов: `kebab-case`.
- Документы этапов: русские названия по структуре `docs/requirements/...`.
- Диаграммы: `.plantuml` + рядом `.png` (исходник и рендер в одной папке).

## 7. Точки входа (Entry Points)
- PDD scanner:
  - `python .tools/pdd/pdd_scan.py --help`
  - `.tools/pdd/pdd-scan` (Linux/macOS)
  - `.tools/pdd/pdd-scan.bat` (Windows)
- PlantUML renderer:
  - `python .tools/plantuml-render/plantuml_render.py --help`
  - `.tools/plantuml-render/plantuml-render` (Linux/macOS)
  - `.tools/plantuml-render/plantuml-render.bat` (Windows)

## 8. Критерии актуальности структуры
- Любая новая утилита добавляется в `.tools/registry.json`.
- Любой новый результат этапа SDD размещается в `docs/requirements/`.
- Любая автоматическая проверка в CI должна ссылаться на команды из `.tools/*`, а не дублировать логику inline.
- Профили интеграции репозиториев хранятся в versioned policy-manifests (`.manifest/profiles/*`).
