## 1. Принципы стека
- Кросс-платформенность разработки (Windows/Linux/macOS) через Python + wrappers (PowerShell/Bash).
- Минимум внешних зависимостей: стандартная библиотека Python для локальных инструментов.
- Разделение логики и оркестрации: логика в `.tools/*`, CI orchestration в GitHub Actions workflows.
- На MVP нет выделенной прикладной БД (Application DB): используется гибрид Git + Checks + Artifacts + Cache.
- Multi-repo интеграция: подключение новых репозиториев через `Repository Integration Profile` и reusable workflows.

## 3. Стек по слоям (Layers)

### 3.1 Прикладной слой (Application Layer)
- Workflow Orchestrator (Оркестратор workflow)
- Policy Resolver (Разрешитель политик)
- Verification Runner (Исполнитель проверок)
- Merge Decision Engine (Движок решения merge)
- Evidence Publisher (Публикатор доказательной информации)

Реализация: scripts-first, с переносом orchestration в `.tools/*` и вызовом из CI/workflows.

### 3.2 Технологический слой (Technology Layer)
- Hosted GitHub runners (Linux/Windows).
- Python runtime + shell wrappers.
- GitHub Checks API, Artifacts, Cache.
- Локальный Kroki в Docker для рендера диаграмм.

## 5. Интеграции и контракты
- Входящие события (Incoming Events): `pull_request`, `push`, `workflow_dispatch` (через GitHub Actions).
- Выходы проверок (Check Outputs): status checks + logs + artifacts.
- Управление внедрением: `enforcement_mode` (`audit|required`) и `rollout_channel` (`pilot|wave|global`) задаются policy/profile слоем.
- Локальные точки запуска:
  - `python .tools/pdd/pdd_scan.py`
  - `python .tools/plantuml-render/plantuml_render.py`

## 6. Риски стека и меры
- Риск: drift версий рантаймов между локальной средой и CI.
  - Мера: фиксировать версии в workflows и документации.
- Риск: перерасход бесплатных CI-минут.
  - Мера: локальный pre-check + кэш + фильтры по измененным путям.
- Риск: зависимость от внешнего endpoint Kroki.
  - Мера: локальный Kroki через Docker Compose.

## 7. Итог
Текущий стек соответствует выбранной архитектуре Layered + Hexagonal (Слоистая + Порты и адаптеры) и модели `local pre-check + PR gates`.

Приоритет следующего шага: внедрить workflows с фиксированными версиями инструментов и required checks в branch protection.
