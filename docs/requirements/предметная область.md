# Предметная область системы RRDCS

## 1. Расшифровка названия / аббревиатуры
**RRDCS** = **Reproducible, Rule-Driven Codebase Shield**

**Суть (1-2 предложения):** RRDCS задает набор обязательных правил и автоматических quality gates, через которые проходят изменения в кодовой базе. Система снижает риск деградации качества кода при активной работе человека и ИИ-агентов за счет воспроизводимых проверок и архитектурных ограничений.

## 2. Обзор предметной области

### 2.1 Контекст проблемы
**Наблюдение/тенденция:** ИИ-ассистенты стали частью ежедневной разработки и массово участвуют в генерации, ревью и рефакторинге кода.

**Боль/проблема:** Код от LLM может выглядеть корректным, но содержать дефекты, нарушения стиля, архитектурные несогласованности и риски безопасности. Без жестких репозиторных барьеров это приводит к накоплению технического долга и снижению устойчивости кодовой базы.

**Почему сейчас:** По мере роста доли ИИ-вклада растет и скорость попадания некачественных изменений в репозиторий. Нужен стандартизованный механизм, который автоматически блокирует merge при нарушении правил качества.

### 2.2 Цель системы
Система предназначена для:
- фиксации единого контракта разработки (правила, паттерны, ограничения) для людей и ИИ;
- автоматической проверки каждого изменения через quality gates в CI;
- контроля архитектурных инвариантов и границ модулей;
- поддержания воспроизводимого уровня качества кодовой базы при масштабировании команды.

**Вне цели (Out of Scope):**
- замена инженерного суждения и code review человека;
- выбор одного универсального языка программирования или единственной CI-платформы;
- разработка бизнес-функциональности продукта, не связанной с quality governance.

### 2.3 Критерии успеха
- **SC-01:** pull request не может быть merged при падении обязательных quality gates (tests/lint/style/architecture checks).
- **SC-02:** для репозитория существует и поддерживается актуальный Agent Coding Manifest с правилами именования, структуры и допустимых паттернов.
- **SC-03:** архитектурные ограничения выражены в автоматических проверках (например, правила зависимостей модулей), а не только в текстовых договоренностях.

### 2.4 Текущие процессы (as-is)

**Описание текущих процессов:**
| Процесс | Описание текущего состояния | Проблемы | Затраты/метрики |
|---------|------------------------------|----------|-----------------|
| Внесение изменений в код | Шаги: 1) разработчик или ИИ предлагает изменение 2) код проходит локальную проверку 3) изменение отправляется на review | качество зависит от индивидуальной дисциплины, нет единого барьера для style/architecture | рост стоимости review при больших diff |
| Code Review | Шаги: 1) reviewer оценивает diff 2) принимает/отклоняет изменения 3) merge после согласования | человеческий review не гарантирует выявление всех архитектурных и стилистических нарушений | риск пропуска дефектов при большом объеме PR |
| Поддержание архитектурных правил | Шаги: 1) правила фиксируются текстом 2) команды трактуют их локально 3) нарушения выявляются постфактум | правила не всегда исполняются автоматически, возможны несогласованные решения между модулями | рост технического долга и затрат на рефакторинг |

**Gap Analysis (as-is → to-be):**
| Боль/проблема | Причина (текущий процесс) | Решение (целевая система) | Связь с UC |
|---------------|----------------------------|---------------------------|------------|
| Некачественные изменения проходят в main branch | Отсутствует обязательный единый набор quality gates | Ввести блокирующие CI-проверки для каждого PR | UC-01 |
| Стиль и паттерны применяются непоследовательно | Нет формализованного контракта правил для человека и ИИ | Вести и обновлять Agent Coding Manifest | UC-02 |
| Архитектурные нарушения обнаруживаются поздно | Архитектурные решения не переведены в автоматически проверяемые ограничения | Реализовать архитектурные инварианты как тестируемые правила | UC-03 |

## 3. Участники домена (Actors / Stakeholders)

| Роль/Актор | Описание | Цели | Боли/риски |
|---|---|---|---|
| Разработчик (Developer) | Инженер, который вносит изменения в код и работает с ИИ-ассистентом | быстро поставлять изменения без деградации качества | ручная проверка не масштабируется, сложность исправления поздних дефектов |
| Технический лидер и архитектор (Tech Lead and Architect) | Отвечает за стандарты, архитектурные границы и эволюцию практик | обеспечить единый инженерный контракт и стабильность архитектуры | рассинхронизация стиля и архитектуры между командами |
| Система непрерывной интеграции (Continuous Integration System) | Автоматический исполнитель quality gates и статических проверок | объективно и воспроизводимо валидировать изменения | неполный набор проверок приводит к пропуску проблем |

## 4. Доменная модель

### 4.1 Глоссарий
| Термин | Определение | Пример | Антипример |
|---|---|---|---|
| RRDCS | Методология и набор репозиторных инструментов для защиты кодовой базы через обязательные правила и автоматические проверки. | Репозиторий блокирует merge, пока не пройдены tests/lint/architecture rules. | Не набор разрозненных рекомендаций без автоматического enforcement. |
| Quality Gate | Обязательная автоматическая проверка, которую изменение должно пройти до merge. | CI job `test+coverage` отклоняет PR при падении порога покрытия. | Не ручная пометка в комментарии review без блокировки merge. |
| Agent Coding Manifest | Документ с правилами стиля, структуры и паттернов, которые обязательны для команды и ИИ-агентов. | Файл правил определяет naming conventions и запрещенные архитектурные зависимости. | Не неформальные договоренности, известные только части команды. |
| Architecture Invariant | Архитектурное ограничение, формализованное как проверяемое правило. | Правило "модуль Source не зависит от Foo" выражено тестом и падает при нарушении. | Не текстовое правило без автоматической проверки в CI. |
| Style Checker | Инструмент статического контроля, который проверяет соответствие кода принятым стилистическим правилам. | ShellCheck валидирует shell scripts, Hadolint проверяет Dockerfile. | Не ad-hoc замечания после merge без системного контроля. |
| Small Pull Request | Процессное ограничение на размер изменения для повышения качества review. | PR содержит ограниченный diff и проверяется без потери контекста. | Не крупный PR на сотни строк с высокой вероятностью пропуска дефектов. |
| Coverage Control | Контроль минимального уровня покрытия тестами как условия приема изменения. | CI отклоняет PR, если покрытие ниже установленного порога. | Не сбор отчета о coverage без влияния на решение о merge. |

### 4.2 Ключевые сущности
| Сущность | Описание | Ключевые атрибуты | Роль в сценариях |
|---|---|---|---|
| Rule Set | Набор обязательных правил проекта (manifest + проверки), определяющий критерии приемки изменений. | `rule_id`, `scope`, `enforcement_level` | Используется в UC-01, UC-02 для валидации и обновления стандартов |
| Pull Request Change | Изменение, которое претендует на попадание в основную ветку и проходит все гейты. | `pr_id`, `changed_files`, `status` | Центральный объект UC-01 |
| Gate Result | Результат выполнения quality gate по конкретному изменению. | `gate_name`, `result`, `evidence` | Формирует решение о допуске/блокировке merge в UC-01 и UC-03 |

### 4.3 Базовые доменные правила
- **BR-01:** Любой Pull Request Change допускается к merge только при успешном прохождении всех обязательных quality gates из Rule Set.
  - **Причина:** предотвращение попадания деградирующих изменений в основную ветку.
- **BR-02:** Agent Coding Manifest обязателен как единый источник правил для участников разработки и ИИ-агентов.
  - **Причина:** сохранение согласованного стиля и архитектурной целостности.
- **BR-03:** Архитектурные ограничения должны быть выражены в проверяемых правилах, а не только в текстовых договоренностях.
  - **Причина:** снижение зависимости от субъективности ручного review.

## 5. Доменные процессы и сценарии (Use Cases)

### 5.1 Карта сценариев
- **UC-01:** Validate Change by Quality Gates — не допустить merge изменений, нарушающих обязательные проверки — актёр: Система непрерывной интеграции (Continuous Integration System)
- **UC-02:** Maintain Agent Coding Manifest — поддерживать единый и актуальный контракт правил разработки — актёр: Технический лидер и архитектор (Tech Lead and Architect)
- **UC-03:** Enforce Architecture Invariants — автоматически обнаруживать нарушения архитектурных границ — актёр: Система непрерывной интеграции (Continuous Integration System)

### 5.2 Детальное описание
#### UC-01: Validate Change by Quality Gates
- **Цель:** определить, может ли Pull Request Change быть merged в основную ветку.
- **Актор:** Система непрерывной интеграции (Continuous Integration System)
- **Триггер:** создание или обновление pull request.
- **Предусловия:** Rule Set определен; для pull request доступен исполняемый pipeline.
- **Основной поток:**
  1. Система непрерывной интеграции (Continuous Integration System) запускает обязательные quality gates для Pull Request Change.
  2. Система непрерывной интеграции (Continuous Integration System) собирает Gate Result по каждому проверяемому правилу Rule Set.
  3. Система непрерывной интеграции (Continuous Integration System) выставляет итоговый статус pull request: `pass` (merge разрешен) или `fail` (merge заблокирован).
- **Данные:** вход — Pull Request Change, Rule Set; выход — набор Gate Result и итоговый статус допуска.
- **Постусловия:** изменение либо допущено к merge, либо возвращено на доработку.

## 6. Риски и допущения
- **A-01 (допущение):** репозиторий допускает обязательные статус-проверки перед merge.
- **A-02 (допущение):** команда согласна использовать единый Agent Coding Manifest как нормативный артефакт.
- **R-01 (риск):** избыточно жесткие гейты без поэтапного внедрения могут увеличить время прохождения PR.
- **R-02 (риск):** неполное покрытие архитектурных инвариантов автоматическими правилами оставит часть нарушений на ручной review.
