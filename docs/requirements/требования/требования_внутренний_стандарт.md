# Требования к системе RRDCS (внутренний стандарт)

**Версия:** 1.0  
**Дата:** 2026-04-03  
**Стандарт:** SDD-манифесты проекта (`.manifest/*`, `.requirements/*`)  
**Источники:** `docs/requirements/предметная область.md`, `docs/requirements/обоснование выбора.md`, `.sources/*`

## 1. Назначение документа

Документ фиксирует функциональные (`FR-*`) и нефункциональные (`NFR-*`) требования к RRDCS как системе обязательных quality gates для pull request в репозитории.

## 2. Контекст и границы

### 2.1 Контекст
- При совместной разработке человеком и ИИ возрастает риск попадания в репозиторий дефектов, нарушений стиля, уязвимостей и архитектурных нарушений.
- Цель RRDCS — не допускать такие изменения в `main` за счет обязательных автоматических проверок.
- Операционная модель состоит из двух уровней:
  - `Локальный pre-check` — быстрые проверки перед `push/PR` в среде разработчика.
  - `PR gates в GitHub Actions` — обязательные проверки в CI при каждом `pull_request`.
- Правило допуска:
  - если хотя бы один обязательный `PR gate` не пройден, `merge` запрещен;
  - если все обязательные `PR gates` пройдены, `merge` разрешен.

### 2.2 Входит в область требований
- Контроль допуска изменений в main/mainline через обязательные проверки.
- Оркестрация проверок кросс-платформенными скриптами и CI workflow.
- Контроль стиля, тестов, security baseline и архитектурных инвариантов.
- Подключение разных репозиториев к RRDCS через единый профиль интеграции репозитория (Repository Integration Profile).

### 2.3 Вне области требований
- Реализация бизнес-функциональности прикладных систем.
- Внедрение распределенной событийной архитектуры (CQRS/Event Sourcing/Pub/Sub) на старте.

## 4. Нефункциональные требования (NFR)

| REQ-ID | Категория | Требование | Приоритет | Критерий приемки | Верификация |
|---|---|---|---|---|---|
| NFR-001 | Надежность | Допуск в main только через required checks. | MUST | 100% merge операций в protected branch происходят при `pass` required checks. | Инспекция |
| NFR-002 | Производительность | Core PR pipeline должен завершаться не более чем за 15 минут (median). | SHOULD | Медианное время завершения core workflow <= 15 мин на интервале наблюдения. | Анализ |
| NFR-003 | Безопасность | Проверка секретов и базовый SAST выполняются для каждого PR. | MUST | На каждом PR выполняются security jobs; их отключение запрещено policy. | Инспекция/Тест |
| NFR-004 | Поддерживаемость | Используются фиксированные версии рантаймов и ключевых toolchain. | MUST | Версии runtime/tools закреплены в workflow/config; плавающие major не используются. | Анализ |
| NFR-005 | Портируемость | Оркестрация проверок должна быть CI-agnostic на уровне scripts. | SHOULD | Основная логика находится в scripts, а workflow выступает оркестратором. | Анализ |
| NFR-006 | Наблюдаемость | Для каждого failed check доступны лог и краткое summary. | MUST | В каждом failed job есть логи и PR summary со ссылкой на шаг причины. | Демо |
| NFR-007 | Масштабируемость | Добавление нового check не требует изменения существующих FR-критериев. | SHOULD | Новый check добавляется как конфигурационный элемент и отдельный шаг/job. | Анализ |
| NFR-008 | Качество тестирования | На этапе MVP coverage для scripts собирается как метрика (non-blocking), далее переводится в blocking-порог. | SHOULD | Есть job сбора coverage и план повышения порога в roadmap. | Анализ |
| NFR-009 | Эксплуатационная простота внедрения | Подключение нового репозитория (Onboarding) в режиме `audit` должно занимать не более 1 рабочего дня при наличии стандартного GitHub PR-потока. | SHOULD | Есть воспроизводимый checklist подключения (onboarding checklist) и успешный прогон T-011. | Демо/Анализ |

## 5. Ограничения и допущения

### 5.1 Ограничения
- Основная CI-платформа на старте: GitHub Actions.
- Формат диаграмм: PlantUML.
- На старте не используются CQRS/Event Sourcing/Pub/Sub.

### 5.2 Допущения
- Репозиторий поддерживает защищенные ветки (branch protection) и обязательные проверки статуса (required checks).
- Команда принимает единый contract правил и использует `check-all` локально.

## 8. Критерии приемки и сценарии проверки

### 8.1 Общие условия приемки
- Для `main` (или целевой защищенной ветки, protected branch) включены `branch protection` и `required status checks`.
- Обязательные проверки выполняются в PR автоматически.
- Локальный `check-all` доступен и документирован для разработчиков.

### 8.2 Минимальные критерии приемки (release gate)
- [ ] Все `MUST` требования `FR-*` реализованы и верифицированы.
- [ ] Все `MUST` требования `NFR-*` реализованы и верифицированы.
- [ ] Нет открытых `HIGH`/`MEDIUM` пробелов в `gap-tracking.md`, блокирующих этап.
- [ ] Для всех required checks определены owner и критерий успеха.

### 8.3 Тест-сценарии верификации

| TEST-ID | Тип | Цель | Шаги | Ожидаемый результат | Покрываемые требования |
|---|---|---|---|---|---|
| T-001 | Негативный | Проверить блокировку merge при fail check | 1) Создать PR с нарушением lint 2) Дождаться CI | PR получает `failed` status, merge недоступен | FR-001, FR-002, FR-006, NFR-001 |
| T-002 | Позитивный | Проверить разрешение merge при pass checks | 1) Исправить нарушение в PR 2) Перезапустить CI | Все required checks `pass`, merge доступен | FR-003, NFR-001 |
| T-003 | Паритет | Проверить соответствие local pre-check и CI core checks | 1) Запустить `check-all` локально 2) Сравнить с результатом PR jobs | Результаты core-checks эквивалентны | FR-004, NFR-005 |
| T-004 | Security негативный | Проверить срабатывание baseline security checks | 1) Добавить тестовый секрет/нарушение 2) Открыть PR | Security job падает, merge блокируется | FR-007, NFR-003 |
| T-005 | Наблюдаемость | Проверить диагностируемость fail-сценариев | 1) Сгенерировать падение check 2) Проверить PR summary/artifacts | Есть лог причины падения и идентификация шага | FR-008, NFR-006 |
| T-006 | Кросс-платформенность | Подтвердить запуск проверок в разных ОС | 1) Запустить CI matrix (`windows`, `linux`) 2) Проверить исходы | Jobs выполняются успешно на обеих ОС | FR-009 |
| T-007 | Расширяемость | Подтвердить подключение нового check без переписывания ядра | 1) Добавить новый check через конфигурацию/адаптер 2) Запустить PR | Новый check выполняется, core orchestration не изменен | FR-010, NFR-007 |
| T-008 | Производительность | Подтвердить целевое время core pipeline | 1) Собрать статистику по N PR 2) Рассчитать median | Median core pipeline <= 15 мин | NFR-002 |
| T-009 | Версионирование toolchain | Проверить фиксированность runtime/tools | 1) Проанализировать workflow/config 2) Проверить версии | Версии зафиксированы, плавающие major отсутствуют | NFR-004 |

### 8.4 Требования к отчетности по проверкам
- Результат каждого required check должен быть представлен в PR UI (success/fail).
- Для каждого `failed` шага должен быть доступен подробный лог.
- Для релизной приемки формируется краткий верификационный отчет:
  - дата прогона;
  - версия правил/конфигурации;
  - список пройденных TEST-ID;
  - список отклонений и принятые корректирующие действия.
