# Программа и методика испытаний RRDCS

## 1. Титульная часть
- Наименование: `RRDCS (Reproducible, Rule-Driven Codebase Shield)`.
- Вид испытаний: приемочные испытания качества PR-gates и инструментального контура.
- Версия/дата: `1.0 / 2026-04-03`.
- Нормативные ссылки (внутренний стандарт):
  - `docs/requirements/требования/требования_внутренний_стандарт.md`
  - `docs/requirements/обоснование выбора.md`
  - `docs/requirements/структура ПО/стек проекта.md`
  - `docs/requirements/архитектура/описание AL.md`
  - `docs/requirements/архитектура/описание TL.md`
  - `docs/requirements/структура Данных/описание Данных.md`

## 2. Общие положения
- Назначение: подтвердить, что `merge` в защищаемую ветку разрешается только при прохождении обязательных quality gates.
- Объект испытаний:
  - in scope:
    - PR-triggered orchestration.
    - Required checks (lint/static, security, matrix, evidence publication).
    - Merge decision и публикация доказательной базы.
  - out of scope:
    - прикладная бизнес-функциональность внешних систем;
    - выделенная прикладная БД (на MVP не используется).
- Основания:
  - FR/NFR и тест-сценарии `T-001 ... T-010` из `требования_внутренний_стандарт.md`, раздел `8.3`;
  - модель слоев и сервисов из `описание AL.md` и `описание TL.md`.

## 3. Приемочные критерии

| ID требования | Текст требования | Значение/порог | Источник |
|---|---|---|---|
| AC-001 | Обязательные проверки запускаются на каждом PR | `pass/fail` факт запуска required jobs | FR-001 |
| AC-002 | Merge блокируется при fail required check | merge недоступен при `failed` | FR-002 |
| AC-003 | Merge разрешается при pass required checks | merge доступен при `pass` | FR-003 |
| AC-004 | Security baseline выполняется на каждом PR | есть security jobs на каждом PR | FR-007, NFR-003 |
| AC-005 | Для fail-check доступны причина и лог | причина + `log_ref` для каждого fail | FR-008, NFR-006 |
| AC-006 | Кросс-платформенность core checks | минимум `windows` и `linux` matrix | FR-009 |
| AC-007 | Core pipeline производительности | median `<= 15 минут` | NFR-002 |
| AC-008 | Runtime/toolchain зафиксированы | нет плавающих major | NFR-004 |
| AC-009 | Coverage режим MVP | coverage собирается non-blocking | NFR-008 |

## 4. Условия проведения испытаний
- Стенд:
  - `GitHub Linux Runner` — исполнение Linux jobs.
  - `GitHub Windows Runner` — исполнение Windows jobs.
  - `Developer Workstation` — локальный pre-check (`check-all`).
- ПО и конфигурации:
  - `Python 3.12 Runtime`.
  - `PowerShell 7+`, `Bash 5+`.
  - `GitHub Actions Runtime`, `GitHub Checks API`, `GitHub Artifacts`, `GitHub Cache`.
- Сетевые условия:
  - для документационных рендеров используется локальный `Kroki` (`localhost:${KROKI_PORT:-8000}`).
- Контроль работоспособности:
  - доступность workflow job logs и PR summary;
  - доступность artifact links для результатов проверок.

## 5. Подготовка к испытаниям
1. Подтвердить включенные branch protection и required checks.
2. Подтвердить наличие policy/manifests в репозитории как source of truth.
3. Подтвердить доступность CI раннеров для Windows и Linux.
4. Подтвердить, что диагностика fail-check публикуется в summary/artifacts.
5. Подтвердить единый локальный pre-check сценарий (`check-all`) для паритета с CI core checks.

## 7. Методики испытаний (по ID проверок)

### T-001 — Блокировка merge при падении lint check
- Цель: подтвердить обязательность lint/static quality gate.
- Основание: FR-001, FR-002, FR-006; NFR-001.
- Средства: PR workflow required checks.
- Шаги:
  1. Создать PR с нарушением lint/static.
  2. Дождаться завершения required jobs.
  3. Проверить статус merge.
- Ожидаемый результат: merge заблокирован.
- Артефакты: job logs, checks status.
- Оценка соответствия: pass/fail.

### T-002 — Разрешение merge при полном pass required checks
- Цель: подтвердить корректный позитивный исход gate-контроля.
- Основание: FR-003; NFR-001.
- Средства: PR workflow required checks.
- Шаги:
  1. Исправить нарушения в PR.
  2. Повторно выполнить required checks.
  3. Проверить доступность merge.
- Ожидаемый результат: merge разрешен.
- Артефакты: checks status, PR summary.
- Оценка соответствия: pass/fail.

### T-003 — Паритет local pre-check и CI core checks
- Цель: подтвердить отсутствие дрейфа локального и CI наборов.
- Основание: FR-004; NFR-005.
- Средства: `check-all` + CI core jobs.
- Шаги:
  1. Запустить локальный `check-all`.
  2. Открыть/обновить PR.
  3. Сравнить результат core checks.
- Ожидаемый результат: статусы эквивалентны.
- Артефакты: local log, CI log, compare report.
- Оценка соответствия: pass/fail.

### T-004 — Срабатывание baseline security checks
- Цель: подтвердить блокировку при security нарушениях.
- Основание: FR-007; NFR-003.
- Средства: security jobs (`gitleaks`, `CodeQL` профиль).
- Шаги:
  1. Подготовить тестовое security-нарушение.
  2. Запустить PR checks.
  3. Проверить merge status.
- Ожидаемый результат: merge blocked.
- Артефакты: findings report, security logs.
- Оценка соответствия: pass/fail.

### T-005 — Наблюдаемость fail-сценариев
- Цель: подтвердить полноту доказательной базы.
- Основание: FR-008; NFR-006.
- Средства: PR summary + GitHub Artifacts.
- Шаги:
  1. Инициировать контролируемый fail одного required check.
  2. Проверить summary и ссылку на logs/artifacts.
  3. Проверить описание причины fail.
- Ожидаемый результат: причина и `log_ref` присутствуют.
- Артефакты: run summary, artifact links.
- Оценка соответствия: pass/fail.

### T-006 — Кросс-платформенный запуск
- Цель: подтвердить выполнение core checks на Windows и Linux.
- Основание: FR-009.
- Средства: matrix CI jobs.
- Шаги:
  1. Запустить pipeline с матрицей `windows+linux`.
  2. Проверить исходы jobs.
- Ожидаемый результат: успех в обеих ОС.
- Артефакты: matrix logs.
- Оценка соответствия: pass/fail.

### T-007 — Подключение нового check без переписывания ядра
- Цель: подтвердить расширяемость orchestration.
- Основание: FR-010; NFR-007.
- Средства: конфигурационное подключение check.
- Шаги:
  1. Добавить новый check в конфигурационный слой.
  2. Запустить PR workflow.
  3. Проверить выполнение нового check и отсутствие изменений core flow.
- Ожидаемый результат: новый check выполняется, core orchestration не переписан.
- Артефакты: change diff, workflow logs.
- Оценка соответствия: pass/fail.

### T-008 — Производительность core pipeline
- Цель: подтвердить целевое медианное время.
- Основание: NFR-002.
- Средства: статистика длительностей workflow.
- Шаги:
  1. Собрать выборку по N PR.
  2. Рассчитать median.
  3. Сравнить с целевым порогом.
- Ожидаемый результат: median `<= 15 минут`.
- Артефакты: performance summary.
- Оценка соответствия: pass/fail.

### T-009 — Фиксация версий runtime/toolchain
- Цель: подтвердить воспроизводимость toolchain.
- Основание: NFR-004, BR-PG-02.
- Средства: audit workflows/config/manifests.
- Шаги:
  1. Проверить runtime/toolchain versions.
  2. Проверить отсутствие плавающих major.
- Ожидаемый результат: версии закреплены.
- Артефакты: pin-audit report.
- Оценка соответствия: pass/fail.

### T-010 — Coverage на MVP (non-blocking)
- Цель: подтвердить режим сбора coverage как метрики.
- Основание: NFR-008.
- Средства: coverage job.
- Шаги:
  1. Запустить coverage job в CI.
  2. Проверить, что результат публикуется.
  3. Проверить, что job informational и не блокирует merge.
- Ожидаемый результат: coverage отчет есть, merge не блокируется этим job.
- Артефакты: coverage report.
- Оценка соответствия: pass/fail.

## 8. Порядок проведения испытаний
1. Проверка готовности условий (раздел 5).
2. Выполнение обязательных негативных/позитивных gate проверок `T-001`, `T-002`, `T-004`.
3. Выполнение наблюдаемости и кросс-платформенности `T-005`, `T-006`.
4. Выполнение паритета и расширяемости `T-003`, `T-007`.
5. Выполнение аналитических проверок `T-008`, `T-009`, `T-010`.
6. Формирование итогового протокола.

Переход между шагами разрешен только после фиксации результатов предыдущего шага.

## 10. Критерии завершения испытаний
- Все обязательные проверки `T-001 ... T-010` выполнены и протоколированы.
- Все критерии `AC-001 ... AC-009` подтверждены.
- Для всех `failed` случаев есть причина и ссылки на evidence.
- Нет блокирующих отклонений, противоречащих FR/NFR `MUST`.

## 11. Работы после завершения испытаний
- Сохранить итоговый протокол и ссылки на artifacts в репозитории.
- Зафиксировать отклонения и корректирующие действия (при наличии).
- Обновить baseline test stack и ПМИ при изменениях policy/toolchain/check-наборов.
