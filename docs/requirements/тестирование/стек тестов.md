## 3. Описание уровней

### 3.1 Линт и статический анализ (Lint and Static)
- Инструменты: `ShellCheck`, `Hadolint`, `markdownlint`, `actionlint`.
- Запуск: через `check-all` локально и required checks в PR.
- Окружение: `Python 3.12 runtime`, wrappers `PowerShell 7+` и `Bash 5+`.
- Артефакты: логи шагов, summary статусов.
- Критерии приемки: отсутствие `failed` по required lint/static checks.
- Владелец: домен `Пакеты проверок (Verification Packages)`.

### 3.2 Базовый уровень безопасности (Security Baseline)
- Инструменты: `gitleaks`, `CodeQL` (базовый профиль).
- Запуск: required security jobs в каждом `pull_request`.
- Окружение: GitHub hosted runners, policy/toolchain pins.
- Артефакты: security logs, findings, summary.
- Критерии приемки: нет блокирующих findings, merge не блокируется security gate.
- Владелец: домены `Пакеты проверок` + `Управление политиками`.

### 3.3 Интеграция и поток гейтов (Integration / Gate Flow)
- Инструменты: orchestration scripts + CI workflows.
- Запуск: события PR, затем check-plan -> checks -> merge decision.
- Окружение: `API pull request в GitHub`, `GitHub Checks API`.
- Артефакты: run summary, merge decision evidence.
- Критерии приемки: соблюдение правил FR-001/FR-002/FR-003.
- Владелец: домен `Оркестрация проверок качества`.

### 3.4 Матрица платформ (Platform Matrix)
- Инструменты: matrix jobs в CI.
- Запуск: минимум `windows` и `linux`.
- Окружение: `TN-001 Linux runner`, `TN-002 Windows runner`.
- Артефакты: matrix logs + итоговые статусы.
- Критерии приемки: core checks успешно выполняются на обеих платформах.
- Владелец: домены `Оркестрация проверок качества` + `Пакеты проверок`.

### 3.5 Наблюдаемость и доказательства (Observability and Evidence)
- Инструменты: PR summary + artifact storage.
- Запуск: публикация после завершения check-run.
- Окружение: `GitHub Artifacts Service`, PR UI.
- Артефакты: summary, ссылки на логи, verification artifacts.
- Критерии приемки: для каждого failed check есть причина и `log_ref`.
- Владелец: домен `Отчетность и доказательная база`.

### 3.6 Производительность основного pipeline (Performance of Core Pipeline)
- Инструменты: метрики длительности workflow.
- Запуск: автоматический сбор длительностей core gate pipeline.
- Окружение: GitHub Actions.
- Артефакты: duration summary/report.
- Критерии приемки: median `<= 15 минут` (NFR-002).
- Владелец: домен `Оркестрация проверок качества`.

### 3.7 Управление политиками и аудит pin-версий (Governance and Pin Audit)
- Инструменты: policy/toolchain validation + pin-audit.
- Запуск: при изменениях policy/workflow/toolchain.
- Окружение: policy manifests в Git.
- Артефакты: validation report, governance decision evidence.
- Критерии приемки: runtime/tools зафиксированы, плавающие major отсутствуют.
- Владелец: домен `Управление политиками и цепочкой инструментов`.

### 3.8 Метрики покрытия (Coverage Metrics, MVP)
- Инструменты: coverage job для scripts/tests.
- Запуск: в CI как отдельный non-blocking job.
- Окружение: GitHub Actions.
- Артефакты: coverage report.
- Критерии приемки: метрика собирается; блокировка merge отключена на MVP.
- Владелец: домен `Пакеты проверок`.
- Примечание для stage-2: coverage проходит путь `M0 -> M1 -> M2 -> M3`, где `M2` соответствует пилотному blocking и использует порог `line coverage >= 55%` согласно `docs/requirements/тестирование/coverage-roadmap.md`.

### 3.9 Подключение нескольких репозиториев и rollout (Multi-Repository Onboarding and Rollout)
- Инструменты: Repository Integration Profile + reusable workflow + governance rollout policy.
- Запуск: mixed (часть шагов в PR/CI, часть по onboarding checklist).
- Окружение: целевой репозиторий + GitHub Actions + policy manifests.
- Артефакты: onboarding checklist, run logs в `audit` и `required`, rollout decision log.
- Критерии приемки: репозиторий подключается без изменения core scripts и успешно переводится `audit -> required`.
- Владелец: домены `Управление политиками` + `Оркестрация проверок качества`.

### 3.10 Политика малых PR (Small PR Policy)
- Инструменты: `build_pr_size_report.py` + PR summary.
- Запуск: на каждом PR как warning-only отчет.
- Окружение: GitHub PR events, summary/artifact publication.
- Артефакты: `pr-size-report.json`, `pr-size-report.md`, PR summary.
- Критерии приемки: code PR с `churn <= 100 changed lines` проходит без предупреждения; code PR выше порога получает warning; docs-only PR классифицируются отдельно и не блокируют merge на MVP.
- Владелец: домен `Управление политиками и цепочкой инструментов`.
- Примечание для stage-2: эта политика может стать кандидатом на pilot blocking только после подтвержденных метрик false positive и согласованного rollback.

## 4. CI/CD: job-профиль и порядок
1. `lint-static` — required.
2. `security-baseline` — required.
3. `core-gate-flow` — required.
4. `platform-matrix` — required для Windows/Linux.
5. `evidence-publish` — required.
6. `pipeline-performance` — informational.
7. `toolchain-pin-audit` — required.
8. `coverage-mvp` — informational (non-blocking).
9. `repository-onboarding-rollout` — mixed (governance + CI evidence).

Политика блокировки merge:
- Любой fail в required jobs блокирует merge.
- Informational jobs не блокируют merge, но фиксируются в отчетности.

## 5. Окружение и тестовые данные
- Основные узлы исполнения:
  - `GitHub Linux Runner`.
  - `GitHub Windows Runner`.
  - `Developer Workstation` (локальный pre-check).
- Локальный сервис диаграмм (для документации): `Kroki` на `${KROKI_PORT:-8000}`.
- Порты прикладных runtime-сервисов на текущем этапе отсутствуют.
- Источники данных:
  - policy/config из Git.
  - checks events из GitHub API.
  - artifacts/summaries из GitHub Artifacts.

## 6. Критичные регрессионные сценарии
- Блокировка merge при fail required check (`T-001`, `T-004`).
- Разблокировка merge после исправления (`T-002`).
- Паритет локального `check-all` и PR core checks (`T-003`).
- Наличие доказательной базы по fail (`T-005`).
- Стабильная работа на Windows/Linux (`T-006`).
- Контроль расширяемости check-набора (`T-007`).
- Управляемый onboarding нового репозитория (`T-011`).

## 7. Метрики и SLA

| Метрика | Целевое значение | Источник |
|---|---|---|
| Медианная длительность core pipeline (Core pipeline median duration) | `<= 15 минут` | NFR-002 |
| Надежность merge-gate (Merge-gate reliability) | merge только при pass required checks | NFR-001 |
| Покрытие базовой security-проверки (Security baseline coverage) | security checks на каждом PR | NFR-003 |
| Полнота наблюдаемости (Observability completeness) | причина + лог для каждого failed check | NFR-006 |
| Режим покрытия (Coverage mode, MVP) | сбор метрики без блокировки merge | NFR-008 |
| Порог small PR policy (Small PR threshold) | `<= 100 changed lines` для code PR; docs-only PR отдельно | policy-and-toolchain-governance |
| Время подключения репозитория (Repository onboarding lead time) | <= 1 рабочий день (в режиме `audit`) | NFR-009 |
