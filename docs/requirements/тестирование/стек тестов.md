## 2. Уровни тестирования

| Уровень | Область | Цель |
|---|---|---|
| Lint and Static | Shell/Dockerfile/Markdown/Workflow | Базовая блокировка ошибок качества до merge |
| Security Baseline | Secret scanning + SAST | Блокировка секретов и базовых уязвимостей |
| Integration / Gate Flow | PR event -> required checks -> decision | Подтверждение контракта gate-оркестрации |
| Platform Matrix | Windows + Linux | Кросс-платформенное исполнение core checks |
| Observability and Evidence | Summary + logs + artifacts | Диагностика причин fail и аудит |
| Performance of Core Pipeline | Core workflow duration | Контроль медианного времени pipeline |
| Governance and Pin Audit | Policy/toolchain pinning | Воспроизводимость toolchain и стабильность |
| Coverage Metrics (MVP non-blocking) | Coverage scripts/tests | Сбор метрик покрытия без блокировки merge |
| Multi-Repository Onboarding and Rollout | Repository integration profile + enforcement mode transition | Простое подключение репозитория и управляемый переход `audit -> required` |

## 3. Описание уровней

### 3.1 Lint and Static
- Tools: `ShellCheck`, `Hadolint`, `markdownlint`, `actionlint`.
- Run: через `check-all` локально и required checks в PR.
- Env: `Python 3.12 runtime`, wrappers `PowerShell 7+` и `Bash 5+`.
- Artifacts: логи шагов, summary статусов.
- Acceptance criteria: отсутствие `failed` по required lint/static checks.
- Owner: домен `Пакеты проверок (Verification Packages)`.

### 3.2 Security Baseline
- Tools: `gitleaks`, `CodeQL` (базовый профиль).
- Run: required security jobs в каждом `pull_request`.
- Env: GitHub hosted runners, policy/toolchain pins.
- Artifacts: security logs, findings, summary.
- Acceptance criteria: нет блокирующих findings, merge не блокируется security gate.
- Owner: домены `Пакеты проверок` + `Управление политиками`.

### 3.3 Integration / Gate Flow
- Tools: orchestration scripts + CI workflows.
- Run: события PR, затем check-plan -> checks -> merge decision.
- Env: `API pull request в GitHub`, `GitHub Checks API`.
- Artifacts: run summary, merge decision evidence.
- Acceptance criteria: соблюдение правил FR-001/FR-002/FR-003.
- Owner: домен `Оркестрация проверок качества`.

### 3.4 Platform Matrix
- Tools: matrix jobs в CI.
- Run: минимум `windows` и `linux`.
- Env: `TN-001 Linux runner`, `TN-002 Windows runner`.
- Artifacts: matrix logs + итоговые статусы.
- Acceptance criteria: core checks успешно выполняются на обеих платформах.
- Owner: домены `Оркестрация проверок качества` + `Пакеты проверок`.

### 3.5 Observability and Evidence
- Tools: PR summary + artifact storage.
- Run: публикация после завершения check-run.
- Env: `GitHub Artifacts Service`, PR UI.
- Artifacts: summary, ссылки на логи, verification artifacts.
- Acceptance criteria: для каждого failed check есть причина и `log_ref`.
- Owner: домен `Отчетность и доказательная база`.

### 3.6 Performance of Core Pipeline
- Tools: метрики длительности workflow.
- Run: автоматический сбор длительностей core gate pipeline.
- Env: GitHub Actions.
- Artifacts: duration summary/report.
- Acceptance criteria: median `<= 15 минут` (NFR-002).
- Owner: домен `Оркестрация проверок качества`.

### 3.7 Governance and Pin Audit
- Tools: policy/toolchain validation + pin-audit.
- Run: при изменениях policy/workflow/toolchain.
- Env: policy manifests в Git.
- Artifacts: validation report, governance decision evidence.
- Acceptance criteria: runtime/tools зафиксированы, плавающие major отсутствуют.
- Owner: домен `Управление политиками и цепочкой инструментов`.

### 3.8 Coverage Metrics (MVP)
- Tools: coverage job для scripts/tests.
- Run: в CI как отдельный non-blocking job.
- Env: GitHub Actions.
- Artifacts: coverage report.
- Acceptance criteria: метрика собирается; блокировка merge отключена на MVP.
- Owner: домен `Пакеты проверок`.

### 3.9 Multi-Repository Onboarding and Rollout
- Tools: Repository Integration Profile + reusable workflow + governance rollout policy.
- Run: mixed (часть шагов в PR/CI, часть по onboarding checklist).
- Env: целевой репозиторий + GitHub Actions + policy manifests.
- Artifacts: onboarding checklist, run logs в `audit` и `required`, rollout decision log.
- Acceptance criteria: репозиторий подключается без изменения core scripts и успешно переводится `audit -> required`.
- Owner: домены `Управление политиками` + `Оркестрация проверок качества`.

## 4. CI/CD: job-профиль и порядок
1. `lint-static` — required.
2. `security-baseline` — required.
3. `core-gate-flow` — required.
4. `platform-matrix` — required для Windows/Linux.
5. `evidence-publish` — required.
6. `pipeline-performance` — informational.
7. `toolchain-pin-audit` — required.
8. `coverage-mvp` — informational (non-blocking).
9. `repository-onboarding-rollout` — mixed (governance + CI evidence).

Политика блокировки merge:
- Любой fail в required jobs блокирует merge.
- Informational jobs не блокируют merge, но фиксируются в отчетности.

## 5. Окружение и тестовые данные
- Основные узлы исполнения:
  - `GitHub Linux Runner`.
  - `GitHub Windows Runner`.
  - `Developer Workstation` (локальный pre-check).
- Локальный сервис диаграмм (для документации): `Kroki` на `${KROKI_PORT:-8000}`.
- Порты прикладных runtime-сервисов на текущем этапе отсутствуют.
- Источники данных:
  - policy/config из Git.
  - checks events из GitHub API.
  - artifacts/summaries из GitHub Artifacts.

## 6. Критичные регрессионные сценарии
- Блокировка merge при fail required check (`T-001`, `T-004`).
- Разблокировка merge после исправления (`T-002`).
- Паритет локального `check-all` и PR core checks (`T-003`).
- Наличие доказательной базы по fail (`T-005`).
- Стабильная работа на Windows/Linux (`T-006`).
- Контроль расширяемости check-набора (`T-007`).
- Управляемый onboarding нового репозитория (`T-011`).

## 7. Метрики и SLA

| Метрика | Целевое значение | Источник |
|---|---|---|
| Core pipeline median duration | `<= 15 минут` | NFR-002 |
| Merge-gate reliability | merge только при pass required checks | NFR-001 |
| Security baseline coverage | security checks на каждом PR | NFR-003 |
| Observability completeness | причина + лог для каждого failed check | NFR-006 |
| Coverage mode (MVP) | сбор метрики без блокировки merge | NFR-008 |
| Repository onboarding lead time | <= 1 рабочий день (в режиме `audit`) | NFR-009 |
