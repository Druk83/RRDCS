# Обоснование выбора для системы RRDCS

## 1. Обоснование выбора направления

### 1.1 Критерии выбора (из предметной области)

| Критерий | Приоритет (H/M/L) | Как проверяем/подтверждаем | Примечание |
|---|---|---|---|
| Надежность/доступность | H | Обязательные status checks в PR, блокировка merge при fail | Базовый контракт качества |
| Производительность | M | Время выполнения pipeline, стабильность выполнения jobs | Критично для скорости обратной связи |
| Безопасность | H | Secret scanning, политика токенов, права workflow | Защита репозитория и цепочки поставки |
| Интеграции/совместимость | H | Совместимость скриптов и workflow с GitHub Actions | Нужна возможность расширения на другие CI |
| Эксплуатация | H | Наблюдаемость по логам workflow и артефактам | Должны быть воспроизводимые результаты |
| Стоимость владения (TCO) | H | Использование free-квоты GitHub Actions + локальный pre-check | Минимизация расходов на старте |
| Экспертиза команды | H | Простота поддержки bash/python scripts и YAML workflows | Учет текущей практики команды |
| Масштабируемость | M | Разделение проверок на отдельные jobs/workflows | Переход к self-hosted/другим CI без переписывания логики |
| Вендор-локин | M | Вынос логики в `.tools/*`, минимизация GitHub-специфичного кода | Портируемость в будущем |

### 1.2 Стандарт документации

| Вариант | Применяется | Обоснование |
|---------|-------------|-------------|
| ГОСТ 19 (ЕСПД) | нет | Нет требования формальной приемки по ГОСТ 19 |
| ГОСТ 34 (АС) | нет | Нет контрактного требования вести документацию по ГОСТ 34 |
| Внутренний стандарт | да | Используются SDD-манифесты проекта (`.manifest/*`, `.requirements/*`) |
| Без стандарта | нет | Для проекта выбран формализованный процесс SDD |

**Выбранный стандарт:** внутренний стандарт (SDD-манифесты проекта)

### 1.3 Формат диаграмм

| Формат | Применяется | Где используется | Обоснование |
|--------|-------------|------------------|-------------|
| PlantUML | да | Диаграммы сценариев и архитектуры | Есть проектный рендерер `.tools/plantuml-render` |
| Mermaid | нет | — | Не выбран как основной формат на старте |
| Draw.io / Excalidraw | нет | — | Не выбран для автоматизированного пайплайна |

**Выбранный формат:** PlantUML

## 2. Обоснование выбора платформы

### 2.1 Рассмотренные варианты

| Вариант | Плюсы | Минусы | Риски/ограничения | Итог |
|---|---|---|---|---|
| On-prem (VM/Bare metal) | Полный контроль инфраструктуры, нет SaaS-квот платформы | Нужны администрирование и сопровождение CI | Операционные издержки с первого этапа | не выбран на старте |
| Cloud (GitHub Actions) | Быстрый старт, встроенная интеграция с PR, простой onboarding | Лимиты free-квоты и ограничения hosted runners | Возможные ограничения по минутам/конкурентности | выбран |
| Hybrid | Гибкость при росте нагрузки, путь к миграции | Сложнее эксплуатация и поддержка | Повышенная сложность операционной модели | отложено |

### 2.2 Принятое решение

**Выбрано:** Cloud-first на GitHub Actions с возможностью расширения

**Основные причины выбора:**
- полный контроль качества на уровне PR через обязательные workflow checks;
- минимальная стоимость и быстрый старт без разворачивания отдельной инфраструктуры;
- совместимость с моделью "логика проверок в scripts, оркестрация в workflows".

**Границы решения (что входит / что не входит):**
- Входит: `.github/workflows/*` + `.tools/*` scripts + обязательные PR gates.
- Входит: локальный pre-check как ускоритель обратной связи перед push.
- Не входит / отложено: запуск собственного SaaS/отдельной CI-платформы.
- Не входит / отложено: миграция на self-hosted runner до появления объективной необходимости.

## 3. Обоснование архитектурного подхода

### 3.1 Масштаб системы

| Масштаб | Характеристики | Типичные решения |
|---------|----------------|------------------|
| **Малая** | 1-3 разработчика, 1 домен, < 10 UC, < 1000 RPS | Монолит, Layered, минимум документации |
| **Средняя** | 3-10 разработчиков, 2-5 доменов, 10-50 UC, 1K-10K RPS | Модульный монолит, DDD light, полная документация |
| **Крупная** | 10+ разработчиков, 5+ доменов, 50+ UC, 10K+ RPS | Микросервисы/EDA, DDD full, строгие контракты |

**Выбранный масштаб:** малая (стартовый этап внедрения)

**Обоснование:**
- команда: стартовая конфигурация разработки документации и проверок;
- домены: фокус на одном домене quality governance;
- нагрузка: низкая/умеренная, без требований high-throughput;
- срок жизни системы: долгосрочная эволюция с поэтапным усложнением.

### 3.2 Драйверы архитектурного решения

| Драйвер | Оценка (L/M/H) | Описание ситуации | Влияние на выбор |
|---------|----------------|-------------------|------------------|
| Разделение ответственности | M | Логика проверки отделена от оркестрации CI | Скрипты в `.tools/*`, pipeline в workflows |
| Масштабируемость | M | Нужно горизонтально добавлять проверки | Разделение на jobs/workflows |
| Отказоустойчивость | M | Важна стабильность pipeline и повторяемость | Явные checks + fail-fast политика |
| Доменная сложность | M | Есть правила, инварианты и процессные ограничения | Требуется формализация методологии |
| Интенсивность взаимодействия | L | Основные взаимодействия синхронные внутри pipeline | Базовая модель без сложной шины событий |

### 3.3 Рассмотренные варианты

| Вариант | Когда подходит | Сильные стороны | Слабые стороны | Риски | Итог |
|---|---|---|---|---|---|
| Локальные проверки без обязательного CI gate | Небольшие pet-проекты | Быстро и дешево | Нет единого enforcement в PR | Проскок дефектов в main | не выбран |
| Только CI-проверки в PR | Команды с дисциплиной review | Централизованный контроль | Дороже по минутам и медленнее обратная связь | Рост времени итерации | не выбран |
| Гибрид: локальный pre-check + обязательный PR gate | Команды, где важны и скорость, и контроль | Быстрый feedback локально + строгий контроль в PR | Нужна синхронизация локальных и CI-команд | Дрейф окружений | выбран |

### 3.4 Принятое решение

**Выбрано:** гибридная модель контроля качества (local pre-check + GitHub Actions PR gates)

**Почему:**
- снижает расход бесплатных CI-минут за счет раннего локального отсева ошибок;
- сохраняет единый и обязательный критерий допуска в main через CI-status.

**Принципы и ограничения:**
- проверка в PR является источником истины для merge;
- логика проверок хранится в scripts (`.tools/*`), а не в ad-hoc шагах;
- workflow только оркестрирует проверки, не дублируя бизнес-логику scripts.

## 5. Технологический стек

### 5.1 Языки и назначение

| Уровень системы | Основные технологии | Назначение/роль | Примечание |
|---|---|---|---|
| Orchestration / Scripts | Python 3.12 | Единая кросс-платформенная orchestration-логика проверок | Основной runtime для `check-all` |
| Shell wrappers | PowerShell 7+, Bash | Удобные обертки запуска в разных ОС | Тонкий слой, без business-логики |
| CI/CD | GitHub Actions | Запуск обязательных PR gates | Cloud-first, расширяемо |
| Docs/Diagrams | Markdown + PlantUML | Документация и диаграммы | Рендер через `.tools/plantuml-render` |

### 5.2 Хранилища и данные

| Назначение | Система | Обоснование | Альтернативы |
|---|---|---|---|
| Источник правил и кода | Git repository | Версионирование и трассируемость | — |
| Артефакты CI | GitHub Actions Artifacts | Хранение отчетов и логов проверок | Self-hosted artifact store (позже) |
| Кэш зависимостей | GitHub Actions Cache | Снижение времени pipeline | Локальный кэш self-hosted runner |
| Операционные данные качества (MVP) | Отдельная прикладная БД не используется | Для MVP достаточно Git + Checks + Artifacts + Cache, без затрат на сопровождение БД | Выделенная OLTP/OLAP БД (этап эволюции при росте аналитических требований) |

### 5.3 Интеграция и обмен данными

| Тип интеграции | Вариант | Где применяется | Почему выбран |
|---|---|---|---|
| Синхронная | GitHub Checks API / status checks | Результаты PR-gates | Базовый механизм допуска merge |
| Асинхронная | Не используется на старте | — | Для MVP не требуется брокер/шина |
| Входящие вызовы | GitHub events (`pull_request`, `push`) | Триггер workflow | Нативно для выбранной платформы |
| Пакетная | Scheduled workflows (опционально) | Ночные/регулярные проверки | Для экономии минут и профилактики дрейфа |

### 5.4 Модель интеграции в разные репозитории

| Элемент | Решение | Зачем |
|---|---|---|
| Профиль интеграции репозитория (Repository Integration Profile) | Versioned policy/manifests с параметрами `required_checks`, `pinned_versions`, `enforcement_mode` | Подключать репозитории без ручной сборки pipeline с нуля |
| Модель внедрения | Двухшаговая: `audit` -> `required` | Снизить риски при первичном подключении |
| Канал распространения (Rollout Channel) | `pilot` -> `wave` -> `global` | Управляемо масштабировать обновления правил на набор репозиториев |
| Откат | Возврат на предыдущую версию профиля (git-tag/commit) | Быстрый rollback без изменения core scripts |
| Точка исполнения | Reusable workflow + `.tools/*` | Единое поведение в разных репозиториях |

## 6. NFR показатели

| Категория | Целевое значение | Как обеспечиваем (направление) |
|---|---|---|
| Надежность | 100% merge только через обязательные PR checks | Branch protection + required checks |
| Производительность | Core-gate pipeline <= 15 минут (median) | Кэширование, path filters, параллельные jobs |
| Безопасность | Секреты и базовый SAST проверяются в каждом PR | gitleaks + CodeQL (минимальный профиль) |
| Масштабируемость | Добавление нового check без изменения core orchestration | Strategy/Adapter подход в scripts |
| Поддерживаемость | Единая команда локальной проверки (`check-all`) | Централизация запуска и параметров |
| Наблюдаемость | Для каждого fail доступен лог и summary в PR | Artifacts + job summary |

## 7. Требования эксплуатации

1. Доступ к настройке workflow и branch protection только у ролей с повышенными правами.
2. Для администраторов репозитория включается MFA.
3. Ретеншн артефактов CI и логов настраивается централизованно (с периодическим пересмотром).
4. Версии рантаймов и ключевых инструментов фиксируются и регулярно обновляются по расписанию.
5. Все обязательные проверки должны иметь локальный эквивалент запуска (через `check-all`) для уменьшения фейлов в PR.

## 9. Вывод

Принято решение использовать: **GitHub Actions + гибридная модель проверок (local pre-check + PR gates) + Layered/Hexagonal + PlantUML + кросс-платформенный orchestration stack**.

**Выбор обусловлен:**
- требованиями к обязательному quality enforcement в PR;
- необходимостью кросс-платформенной разработки и воспроизводимого исполнения;
- ограничениями по стоимости на старте и возможностью эволюционного расширения;
- выбранной методологией `DDD light + Event Storming + Context Map`.

**Следующий шаг:** этап [3] — формализация требований (FR/NFR) на основе зафиксированного направления.
