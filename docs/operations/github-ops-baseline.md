# Базовый операционный контур GitHub (GitHub Ops Baseline)

## 1. Назначение
Документ фиксирует минимальные операционные настройки GitHub для запуска RRDCS в целевом репозитории.
Это эксплуатационный слой, который дополняет кодовую реализацию `.tools/*` и `.github/workflows/*`.

## 3. Базовые настройки репозитория

### 3.1 Защита ветки (Branch Protection)
- Включить защиту ветки для целевой ветки (`main` или эквивалент).
- Запретить прямой `push` в защищенную ветку.
- Разрешить изменения только через `pull request` (PR).
- Включить требование прохождения проверок статуса перед слиянием (Require status checks to pass before merging).
- Включить требование актуальности ветки перед слиянием (Require branches to be up to date before merging).
- Включить требование закрытия всех обсуждений перед слиянием (Require conversation resolution before merging).

### 3.2 Обязательные проверки (Required Checks)
- Обязательные проверки определяются policy-профилем (policy profile) репозитория.
- Для tooling-репозитория RRDCS используется smoke-план по умолчанию.
- Для целевых репозиториев проверки запускаются через `.github/workflows/rrdcs-reusable-gates.yml` по профилю репозитория (repository profile).
- Режим `audit|required` управляется полем `enforcement_mode` в профиле.
- На MVP минимальный набор:
  - `rrdcs-gate-decision` (обязательный шлюз слияния, merge-gate)

Рекомендуемый дополнительный контроль (может быть необязательным, но видимым в PR):
- `rrdcs-check-all-linux`
- `rrdcs-check-all-windows`

### 3.3 Права доступа (Permissions)
- Принцип минимально необходимых прав (least privilege) для `GITHUB_TOKEN`.
- Для jobs, не публикующих артефакты или комментарии: `contents: read`.
- Для jobs, публикующих доказательства (evidence) в PR: явно задать только необходимые права (`pull-requests: write`, если нужно).
- Доступ к настройкам защиты ветки и обязательных проверок только у ролей `Admin/Maintainer` (по политике организации).

### 3.4 Секреты и переменные (Secrets and Variables)
- Секреты хранятся только в секретах GitHub (GitHub Secrets) на уровнях Repository/Environment/Organization.
- В репозитории запрещен hardcode ключей, токенов и паролей.
- Для каждого секрета должны быть:
  - владелец (`owner`)
  - назначение
  - дата ротации
  - политика ротации

## 4. Чек-лист подключения (Onboarding Checklist)
- Создан профиль интеграции репозитория (Repository Integration Profile).
- Подключен переиспользуемый workflow RRDCS (reusable workflow).
- Настроены обязательные проверки в защите ветки.
- Проверен режим `audit` на контрольном PR.
- Подтвержден переход в `required` после стабилизации.

## 5. Управление изменениями
- Все изменения операционных настроек фиксируются через запрос на изменение (change request) и ссылку на PR/issue.
- Изменение обязательных проверок выполняется синхронно с обновлением policy/profile (политики/профиля).
- При инциденте допускается временный возврат в `audit` с обязательной фиксацией причины и срока восстановления.

## 6. Что не входит в этот документ
- Реализация кода проверок и логики оркестрации (orchestration logic).
- Детальный процесс принятия решения о релизе (release decision process) (см. `docs/release/mvp-release-plan.md`).
