Эта система (RRDCS), включающая в себя набор инструментов, практик и требований для защиты программного кода от негативного влияния искусственного интеллекта (ИИ) и поддержания его высокого качества, не имеет единого конкретного названия, упомянутого в источниках.

Однако, исходя из целей, которые она преследует, ее можно охарактеризовать как **Комплекс мер или систему автоматизированного контроля качества и соответствия стандартам кодирования и архитектуры**.

Основные компоненты этой системы, которые призваны создать **"барьеры"** для низкокачественного кода, генерируемого LLM-ками:

1.  **Контроль качества через покрытие тестами (Test Coverage Control):** Настройка системы сборки для **сбора информации о покрытии кода тестами** и использование этой информации для **отклонения (reject) pull request'ов**, если покрытие падает ниже установленного порога.
2.  **Контроль стилистики (Style Checking):** Использование **максимально жестко настроенных** инструментов, таких как **ClangTidy, ShellCheck или Hadalint**, которые **"бьют по рукам программистов"** (или роботов) за нарушение стилистики и форматирования.
3.  **Документарные стандарты:** Создание **Манифеста стандартов кодирования (Agent Coding Manifest)**, который описывает все принципы программирования проекта для обучения LLM-ки.
4.  **Архитектурный контроль:** Использование таких механизмов, как **ArcUnit**, для формулирования **архитектурных правил** (например, запрета обращения из одного модуля в другой) в виде юнит-теста.

Внедрение всех этих мер гарантирует, что роботы будут **вынуждены выдерживать заданную стилистику и архитектуру**, что позволит приглашать их в команду в большом объеме, не опасаясь, что они навредят.

Если бы эта система была предметом торговли, её можно было бы назвать **Системой защиты качества кодовой базы от агентов ИИ** или **"Роботоустойчивой" системой контроля разработки (Robot-Resistant Development Control System)**.

Для защиты от негативного влияния искусственного интеллекта (ИИ) на качество программного кода можно создать и использовать ряд инструментов и практик, которые будут контролировать стиль, качество и соответствие архитектуре проекта,,.

К основным инструментам и механизмам, которые можно внедрить для защиты, относятся:

### Инструменты для контроля качества и стиля

1.  **Контроль покрытия тестами (Test Coverage Control)**
    Необходимо настроить систему сборки проекта для **сбора информации о покрытии кода тестами (test coverage)**,. Эта информация должна использоваться для контроля качества: если покрытие исходного кода тестами (выраженное в процентах) падает ниже установленной границы, система должна отклонять (reject) pull request, заставляя программиста (или робота) добавить тесты. Идеальное покрытие — 100%, но компания Google, например, использует пять уровней порогов для добровольной системы оповещения,,.
2.  **Инструменты для проверки стилистики (Style Checkers)**
    Необходимо настроить инструменты, которые **«бьют по рукам программистов»** за нарушение стилистики. Эти инструменты должны быть настроены максимально жестко, чтобы LLM-ка натыкалась на барьер и сама разбиралась, как правильно форматировать код, не требуя ручных объяснений от человека,.
    Примеры инструментов для проверки стиля, упомянутые в источниках:
    *   **ClangTidy**, в котором насчитывается почти 700 правил.
    *   **ShellCheck** — чекер для башевских скриптов,.
    *   **Hadalint** — чекер для более грамотного и качественного форматирования Docker-файлов.

### Документарные и архитектурные инструменты

1.  **Манифест стандартов кодирования (Agent Coding Manifest)**
    Это документ, в котором излагаются **все стандарты и принципы программирования**, принятые в проекте. Он служит для обучения LLM-ки, поскольку она не знает вашего стиля и приходит в репозиторий «с чистого листа»,. Вы должны описать в этом документе, как именно вы пишете, какие паттерны используете, а какие нет, и как именуете элементы,,. Для LLM-ки этот манифест необходимо составить на английском языке.
2.  **Описание архитектуры и ArcUnit**
    Для защиты от дефектов архитектурного уровня, когда, например, ошибки в разных частях программы обрабатываются неконсистентными способами,, необходимо изобрести или использовать **язык описания архитектуры**, который LLM-ка могла бы прочитать,.
    *   В качестве временного решения можно использовать хорошо написанный **Readme-файл** для описания архитектуры, так как LLM-ка будет стараться выдать ответ, минимально отклоняясь от контекста, который ее окружает (включая Readme),.
    *   **ArcUnit** — это инструмент, который позволяет сформулировать архитектурные правила (например, запрет на обращение из одного модуля в другой) в виде юнит-теста. Если такое обращение происходит, юнит-тест падает, контролируя соблюдение архитектуры,,.

### Процессные меры, усиливающие контроль

Хотя прямо не являются инструментами, следующие практики необходимы для эффективного использования вышеперечисленных средств:

*   **Код-ревью малыми порциями (Small Pull Requests):** Чтобы иметь возможность проверить код, сгенерированный ИИ, команда должна научиться делать код-ревью небольшими изменениями. Чем меньше кода в pull request, тем качественнее можно проверить работу робота-самозванца, который может вставлять бракованный код,. Желательно, чтобы pull request содержал менее 100 строчек кода.

Если все эти меры (описанная архитектура, настроенные стайл-чекеры и контроль покрытия) будут внедрены, можно будет приглашать роботов в команду в большом объеме, не опасаясь, что они навредят, поскольку они будут вынуждены выдерживать заданную стилистику и архитектуру.
