Определение:
**RRDCS (Reproducible, Rule-Driven Codebase Shield)** — это методология и набор репозиторных инструментов, которые защищают кодовую базу от деградации (в том числе при активном использовании ИИ) за счёт воспроизводимых правил и обязательных автоматических проверок.

**Суть:** любые изменения в коде допускаются только если они проходят заранее заданные “гейты качества”.

**Что входит в RRDCS (ядро):**

* **Манифест правил разработки**: единый стандарт стиля, паттернов, структуры, запретов.
* **Quality Gates в CI**: тесты, проверки регрессий, контроль покрытия и др.
* **Статический контроль**: линтеры/форматтеры/стайл-чекеры с жёсткой конфигурацией.
* **Архитектурные инварианты**: ArcUnit-подобные правила зависимостей и границ модулей.
* **Процессные ограничения**: требования к размеру/структуре PR и чек-листы ревью.

**Результат:** система не “просит верить”, а **принуждает** к соблюдению стандартов — человеку и ИИ одинаково.

Вопрос1: - Предметная область проекта RRDCS?
1. **Предметная область RRDCS**
   Да: это **система/методология/набор инструментов**, которые **защищают кодовую базу от деградации (в т.ч. при активном использовании ИИ)** за счёт обязательных правил и автоматических “quality gates”.

2. **Основные компоненты в центре внимания**

* **Тесты** (включая контроль покрытия/регрессий)
* **Стайл-чекеры / линтеры / форматирование** (строгие правила, автоматическая проверка)
* **Манифест правил разработки** (единые нормы, паттерны, запреты)
* **Архитектурный контроль (ArcUnit-подобные правила)** (зависимости модулей, запреты на связи, архитектурные инварианты)
* **CI-гейты** и политика маленьких PR как поддерживающий механизм

3. **Домены**
   Готового доменного разбиения “как в DDD” в источниках нет — его **нужно вывести** из назначения системы. Минимально разумные домены:

* Governance/Правила репозитория (manifest, contribution rules)
* Quality Gates/Качество (tests, coverage, CI gates)
* Style & Static Analysis/Стиль (lint/format)
* Architecture Compliance/Архитектура (ArcUnit rules)
* Change Control/Процесс изменений (PR policy, review checklists)

Вопрос2: - Целевая аудитория документации?
Целевая аудитория документации RRDCS — **для всех сразу**, но **разделённая по ролям**, чтобы каждый читатель находил свой “слой” без перегруза.

* **Разработчики (внедряющие и работающие в репозитории)**
  Им нужны: как поставить/запустить проверки локально, как чинить типовые падения линтера/тестов, правила PR, ссылки на манифест.

* **Архитекторы/техлиды (проектирующие “защиту”)**
  Им нужны: модель доменов/границ, архитектурные правила (ArcUnit), политика quality gates, уровни строгости и как их эволюционировать.

* **DevOps/Build-инженеры (поднимающие CI/CD)**
  Им нужны: описание пайплайна, матрицы запусков, кеши, артефакты отчётов (coverage), обязательные проверки на merge.

Практическая формулировка: **один комплект документации, три “трека” чтения** (Dev / Arch / CI).

Вопрос3: - Стек технологий?
### Стек технологий (в универсальной модели RRDCS)

**На какие языки ориентироваться?**
На **все**, но через профили:

* **Core (общий для любых языков):**
  форматирование, базовый линт, тесты, отчёт о покрытии/качестве, политика PR, единый манифест правил.

* **Language profiles (подключаемые модули):**

  * **JVM (Java/Kotlin):** профиль с ArcUnit/байткод-анализом для архитектурных правил.
  * **Python:** профиль с pytest + линтер/форматтер + типизация (по потребности проекта).
  * **C/C++:** профиль с clang-format/clang-tidy + тест-раннером (ctest и т.п.).
  * **JS/TS:** профиль с eslint + тест-раннером (jest/vitest и т.п.).
  * **Shell/Dockerfile:** отдельные профили (ShellCheck, Hadolint).

Идея: **RRDCS не выбирает один язык**, он задаёт **контракт гейтов**, а язык определяет “чем именно” эти гейты исполняются.

**Какие CI/CD системы упоминаются?**
В универсальной модели: **любые**, тоже через профили/адаптеры:

* **CI Core (общий контракт):**
  набор обязательных джобов/статусов (format/lint/test/coverage/arch-rules), правила блокировки merge, публикация артефактов (отчёты).

* **CI adapters (реализации):**
  GitHub Actions / GitLab CI / Jenkins / TeamCity / Azure DevOps и т.д. — это разные “обёртки” над одним и тем же набором шагов и выходных статусов.

Итого: “универсальный стек” в RRDCS — это **универсальные гейты + профили реализации** (по языку и по CI), а не один фиксированный список инструментов.

Вопрос5: - Особенности?
* **RRDCS сейчас — скорее концепция/методология**, которую нужно оформить в документации как “шаблон внедрения” (набор обязательных гейтов и правил), а не как один готовый продукт “установил и работает”.

* **При этом отдельные компоненты уже существуют и применяются точечно**:

  * есть пример **agent coding manifest “из нашего репозитория”** (то есть документ реально используется где-то в практике), 
  * **ArcUnit пробовали использовать “кое-где”**, но признаётся, что масштабировать правила сложно и “серьёзных решений пока нет”, 
  * чекеры/coverage/CI-гейты описаны как меры, которые “можно внедрить” и которые должны блокировать PR при нарушениях.

Итого: **не “готовая система с кодовой базой”, а оформляемая методология + набор практик/артефактов**, которые внедряются в конкретные репозитории.
